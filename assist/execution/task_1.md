# Task 1 - Tenant Header Pivot And Platform Gap Closure

## Purpose

This task is a handoff from the senior developer to the next CODEIT agent.

Read `assist/execution/task.md` first, but do not execute it blindly. The plan is broadly suited to the current CODEXSUN codebase, but the framework/platform code has already advanced and one direction must change:

- Do not continue the `x-correlation-id` middleware plan.
- Use `x-tenant-id` as the cross-cutting request context header.
- Keep `requestId` as the technical trace value generated by Fastify.
- Do not generate a tenant id. Tenant id is business context and must come from a validated header, an authenticated session/JWT, or tenant lookup.

The goal of this task is to refactor the current `correlationId` request-context path into a `tenantId` request-context path. This is not an additive change. Remove the remaining correlation-id behavior from API envelopes, request logging, platform routes, events, queues, and tests, then replace the valid business context with `tenantId`.

## Header Contract To Implement

Use this contract unless a human changes the product decision.

| Item | Contract |
| --- | --- |
| Request header | `x-tenant-id` |
| Response header | Echo `x-tenant-id` only when the backend has a known validated tenant id |
| Envelope meta | Always include `requestId` and `timestamp`; include `tenantId` when known |
| Removed API header | `x-correlation-id` |
| Trace value | Use Fastify `request.id`; do not expose or accept a separate correlation header |
| Auth rule | `x-tenant-id` alone never authenticates or authorizes a request |
| Tenant rule | For protected tenant routes, header tenant id must match authenticated tenant context |

Important: current login flow starts with `tenantCode`, not database tenant id. Keep tenant login body as `tenantCode` for now. During successful tenant login, resolve the tenant record and include both `tenantId` and `tenantCode` in the returned session/JWT payload. Subsequent tenant API calls should send `x-tenant-id`.

Required changes:

- Treat this as a refactor from `correlationId` to `tenantId`, not a parallel metadata addition.
- Remove `correlationId` from `FastifyRequest`.
- Remove `x-correlation-id` request parsing and response header output.
- Keep `Permissions-Policy` behavior if still needed, but put it in a generic hook or leave it in `createApiApp`.

- Update request logging to log `requestId`, route, status, duration, and `tenantId` when present.

- If a trace link is still needed for events/jobs, use optional `requestId?: string`, not `correlationId`.

Tests to add/update:

- `x-correlation-id` is ignored and not echoed.
- Error envelopes include `tenantId` when `x-tenant-id` is sent.
- Health route includes tenant meta and still returns `503` only for aggregate `down`.
- Request logging test can be lightweight if logger capture is already difficult; do not overbuild.

Required changes:

- Extend session/JWT payloads to include `tenantId?: string` and `tenantCode?: string`.

- Stop writing tenant code into envelope `tenantId`.

- Keep `tenantCode` in response data where UI needs to show or route by code.
- Add `AUTH_MODE` to API env validation using the existing `authModeSchema`.
- Decide and implement the smallest safe mode behavior for this batch:
  - `jwt`: current bearer-token path.
  - `cookie`: cookie session path.
  - `hybrid`: both cookie and JWT where needed.
- Wire `DatabaseSessionStore` if `cookie` or `hybrid` is active.
- Make logout actually invalidate the active session for cookie/session mode.
- If cookies are implemented, set and clear `codexsun_session` with matching flags.
- Keep routes thin: validation and HTTP mapping in routes, auth logic in `AuthService`.
- Move tenant CRUD route logic out of `auth/routes.ts` or split it into a clearly named tenant route module.
- Add missing super-admin auth guard to `GET /tenants/:id`.
- Protect every tenant management route consistently.
- Validate tenant status and database resolution before tenant login succeeds.

Tests to add/update:

- Successful tenant login returns `tenantId` and `tenantCode`.
- Tenant login fails when tenant code is missing, unknown, inactive, or has no ready database.
- Session check returns `tenantId` for tenant sessions.
- Logout invalidates session where applicable.
- Super admin can list/read tenants.
- Non-super-admin cannot list/read/create/update/delete tenants.
- `GET /tenants/:id` requires auth and super-admin role.

## Acceptance Criteria

- All former `correlationId` request-context usages are either removed or intentionally replaced by `tenantId` or `requestId`, depending on meaning.
- No API response includes `meta.correlationId`.
- No API response header includes `x-correlation-id`.
- `x-tenant-id` can flow into framework request context and envelope meta.
- Protected tenant-scoped routes validate tenant identity against authenticated context.
- Tenant login returns enough information for web to send `x-tenant-id` later.
- Super-admin tenant routes are consistently protected.
- Framework and platform tests cover the header contract and auth gaps.
- Typecheck and lint pass for:
  - `@codexsun/framework`
  - `@codexsun/platform`
  - `@codexsun/platform-api`
  - `@codexsun/platform-web`
- Relevant docs are updated.
